# PDF 导出当前状态与剩余问题

最后更新：2026-03-23

## 当前结论

本轮已把 PDF 导出重新切回“前端导出页渲染完成后，整页截图写入 PDF”的路径。
随后又继续补了两点：

- 导出 PDF 的物理页尺寸不再一律固定，而是开始按原页比例扩展
- 导出分页从容量估算改为真实 DOM 高度测量

当前三条链路的关系如下：

- 前端阅读区：
  使用 `markdown-it + markdown-it-dollarmath + KaTeX`
- HTML 导出：
  继续复用这套前端 HTML / KaTeX 渲染
- PDF 导出：
  先生成与 HTML 导出一致的导出页，再离屏截图成整页 PNG，交给后端写入 PDF

这意味着：

- PDF 不再重新按字符排版讲解文本
- PDF 中公式观感会尽量贴近前端阅读区和 HTML 导出
- 后端 `reportlab` 目前主要承担“把整页图写进 PDF 文件”的职责

## 为什么这次改成图片页

前一轮的核心问题不是“字符顺序有一点点不对”，而是：

- 前端和 PDF 用的是两套完全不同的渲染引擎
- 前端有 KaTeX
- 后端文本排版没有真正的数学布局能力

既然导出页 HTML 本身已经能正确显示公式，那么直接把导出页变成图片页，再写入 PDF，是当前最短路径。

这样做的直接收益是：

- 公式样式与前端更一致
- 不需要继续在后端硬补 LaTeX 文本降级规则
- PDF 观感更接近“看到什么就导出什么”

## 当前已知代价

切到图片页导出后，也带来新的边界：

- PDF 中文字通常不再可直接选中、复制、搜索
- 导出文件体积通常会比纯文本排版更大
- 导出质量依赖截图分辨率与浏览器截图能力
- 当前分页仍然依赖前端的导出页切分逻辑，而不是浏览器原生打印分页

也就是说，现在解决的是“公式观感一致性”问题，不是“一切导出问题都彻底消失”。

## 剩余风险点

当前最需要继续关注的，不再是字符重排，而是下面这些点：

1. 真实高度分页在超大块内容上是否足够稳
   - 超长列表
   - 超长段落
   - 大块展示公式
   - 表格 / 代码块

2. 单个 block 本身过高时是否仍会被裁切
   - 如果表格 / 代码块 / 单个展示公式本身超过一栏高度，截图仍可能裁掉超出部分

3. 大批量导出时的性能与体积
   - 页数多时，前端需要生成大量截图
   - PDF 体积可能明显上升

## 下一步建议

如果继续优化，优先级建议如下：

### 1. 先做导出页溢出检测

在前端生成每张导出页截图前，检查：

- `.text-surface` 是否发生纵向溢出
- 是否有块内容被裁切

如果发现溢出：

- 要么进一步拆分 block
- 要么提示本页导出需调整

### 2. 再做基于真实 DOM 高度的分页

当前切分仍然主要依赖 block + 容量估算。

后续更稳的方向是：

- 先把 block 渲染到离屏 DOM
- 依据真实高度决定这一栏能放多少内容
- 再生成最终截图

这样会比纯字符容量估算更接近真实页面效果。

### 3. 视情况保留双模式

如果后续确实遇到“图片 PDF 不方便搜索/复制”的问题，可以考虑长期保留两种导出模式：

- 图片优先模式：
  公式效果最好，最接近前端
- 文本优先模式：
  便于搜索复制，但公式效果较弱

## 当前判断

对这个项目现阶段来说，把 PDF 导出改为“前端导出页截图入 PDF”是正确方向。

现在的主问题已经从：

- “后端怎么把公式字符近似排出来”

转成了：

- “前端导出页怎么更可靠地分页、避免溢出和裁切”

这比继续在后端做字符级数学补丁更值得投入。
